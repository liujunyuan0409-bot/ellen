# 素材库 Bitable 表字段定义

本文件定义「小红书内容调研素材库」的 Bitable 表结构。表以**一行一篇笔记**为 grain。字段名与 `analysis-rubric.md` 保持一字不差；`采集方式` 枚举与 `gathering.md` 保持一致。

---

## 字段分组总览

字段按以下四个分组排列，与 spec §4.1 完全一致：

1. **采集元数据** — 笔记来源与基本属性
2. **共用打标** — 图文/视频通用分析维度
3. **图文专属** — 仅图文笔记填写，视频笔记留空
4. **视频专属** — 仅视频笔记填写，图文笔记留空

---

## 一、采集元数据

| 字段名 | Bitable 字段类型 | 是否必填 | 说明 |
|--------|----------------|---------|------|
| 笔记链接 | URL | 必填 | 笔记详情页完整 URL，同时作为主键去重 |
| 笔记ID | 文本 | 必填 | 从 URL 中提取的笔记唯一标识符 |
| 账号名 | 文本 | 必填 | 发布笔记的账号昵称 |
| 粉丝量级 | 文本 | 选填 | 账号粉丝数量级别（如 1w以下/1-10w/10-100w/100w+）；Playwright 自动采集时留空，手动/导入补充 |
| 发布日期 | 日期 | 必填 | 笔记发布日期，格式 YYYY-MM-DD |
| 笔记类型 | 单选 | 必填 | 取值：图文 / 视频；决定图文专属/视频专属字段是否填写 |
| 调研主题 | 文本 | 必填 | 本次调研的主题关键词（如「日本自由行攻略」），同一批次笔记填写相同值 |
| 采集方式 | 单选 | 必填 | 取值枚举：自动 / 手动 / 导入（定义见 `gathering.md`） |

---

## 二、共用打标

图文和视频笔记均需填写以下字段。打标在 ANALYZE 步骤由人工或 AI 补充，采集阶段可留空。

| 字段名 | Bitable 字段类型 | 是否必填 | 说明 |
|--------|----------------|---------|------|
| 人设类型 | 单选 | 选填 | 取值：素人路人 / 专业达人 / 品牌官方 / KOC测评党 / 生活方式博主 |
| 选题角度 | 单选 | 选填 | 取值：测评 / 攻略 / 避坑 / 种草 / 教程 / vlog / 情绪共鸣 / 对比 |
| 标题公式 | 单选 | 选填 | 取值：数字清单 / 悬念留白 / 痛点直击 / 身份代入 / 对比反差 / 提问 / 夸张结果 |
| 首句钩子 | 单选 | 选填 | 取值：利益承诺 / 制造焦虑 / 反常识 / 直接提问 / 场景代入（图文=首句；视频=前3秒） |
| 封面形式 | 单选 | 选填 | 取值：大字报 / 实拍单图 / 拼图九宫 / 前后对比 / 人像出镜 / 纯文字卡 |
| 标签策略 | 文本 | 选填 | 记录标签数量及类型组合，如「8个：大词×2 + 长尾×4 + 话题×2」 |
| 赞 | 数字 | 选填 | 点赞数，原始数值 |
| 藏 | 数字 | 选填 | 收藏数，原始数值 |
| 评 | 数字 | 选填 | 评论数，原始数值 |
| 分享 | 数字 | 选填 | 分享数，原始数值；部分笔记不公示则留空 |
| 互动结构 | 文本 | 选填 | 填写赞藏比（藏÷赞）和评论率（评÷赞）；可由 Bitable 公式列自动计算 |
| 爆款归因 | 文本 | 选填 | 一句话说明这篇笔记为何火；无法归因则留空，不编造 |

---

## 三、图文专属

仅图文笔记填写。**视频笔记此字段留空。**

| 字段名 | Bitable 字段类型 | 是否必填 | 说明 |
|--------|----------------|---------|------|
| 内容结构(图文) | 单选 | 选填 | 取值：总分 / 编号清单 / 故事叙事 / step-by-step / 测评打分 |

---

## 四、视频专属

仅视频笔记填写。**图文笔记此组全部字段留空。**

| 字段名 | Bitable 字段类型 | 是否必填 | 说明 |
|--------|----------------|---------|------|
| 内容结构(脚本) | 文本 | 选填 | 描述脚本结构，如「开场钩子 → 痛点 → 解决 → 引导」 |
| 视频时长 | 文本 | 选填 | 秒数或时长区间，如「60s」「3-5min」；从视频页面标注获取 |
| 拍摄形式 | 单选 | 选填 | 取值：口播 / vlog / 教程录屏 / 剧情 / 混剪 |
| 节奏元素 | 多选 | 选填 | 取值（可多选）：BGM / 卡点 / 字幕条 / 花字 |
| 完播信号 | 文本 | 选填 | 对时长与信息密度的推断，如「时长短+信息密=完播率高」；需人工判断 |

## 写入约定

### 建表

- 若素材库 Bitable 表**尚不存在**，使用 `bitable_v1_appTable_create` 创建表，并按上述字段顺序和类型逐一建列。
- 字段名须与本文件字段名**字节一致**（中文字符完全相同，括号使用半角括号 `()`），不得修改或简化。

### 写入行

- 每采集/分析完一篇笔记，使用 `bitable_v1_appTableRecord_create` 追加一行。
- 图文笔记：「视频专属」分组下的全部字段（内容结构(脚本)、视频时长、拍摄形式、节奏元素、完播信号）**保持空值**，不填写占位文字。
- 视频笔记：「图文专属」字段（内容结构(图文)）**保持空值**。
- 任何采集阶段无法获取的字段，统一留空，不推测、不编造。

### CLI / 工具能力校正（写入前确认）

不同 lark-cli / MCP 版本能力不一，按当前环境实测，别假设：
- **批量写入可能不存在**：若无 `record-batch-create`，用 `record-upsert` 逐条写（脚本里循环，加 ~150ms 间隔防限流）。
- **建表 `--fields` 可能不生效**：`base-create` 不一定支持初始字段；必要时先建空表再逐个 `field-create`。
- **单元格格式**：日期=毫秒时间戳（数字）；URL=纯字符串即可；`record-upsert` 的 body 是**裸字段 map**，不要包 `{"fields": ...}`。
- 写入失败先看返回 code（批量超限、并发冲突等）按错误恢复，不要盲目重试。
